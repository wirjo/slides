Design philosophy

Why wrap, not plugin?

You could add an AWS plugin to an existing agent on your laptop. Lowkey takes the opposite approach: deploy the agent inside AWS. Here's why that's better.

✗ Plugin approach
Identity
Agent lives on your laptop/VPS, reaches into AWS via access keys. Keys can leak. Identity is split.
Network
Outside the VPC. Can't access private subnets, internal ALBs, or VPC endpoints. Every API call crosses the internet.
Blast Radius
Single IAM role — hard to scope. Too broad or too narrow. Shared with other tools on the same machine.
Observability
Agent creates resources but can't easily tail logs, port-forward, or SSM into what it built. Debugging loops are slow.
Security
GuardDuty, Security Hub configured separately. No co-deployed watchdog layer. Alerts go somewhere else.
Teardown
Resources accumulate across accounts. No clean "undo." State scattered.
✓ Wrap approach (Lowkey)
Identity
Agent is an AWS identity — EC2 instance role via IMDS. No keys to leak. No credentials in environment variables.
Network
Inside the VPC. Can deploy a service and immediately curl it on private DNS. Sees what it builds.
Blast Radius
The account itself is the boundary. Profile-based IAM + deny policies + account isolation = three layers of defense.
Observability
Runs where the infra is. Tail CloudWatch logs, SSM into containers, check health endpoints — all on localhost or private network.
Security
Security services deploy in the same CloudFormation stack. Agent is watched from minute one. Alerts stay local.
Teardown
One account = one agent. uninstall.lowkey.run destroys the stack. Fresh deploy in 10 minutes.
"It's not an agent with AWS access bolted on.
It's an AWS environment with an agent living inside it."
Security model

Secure despite autonomy

How is an autonomous agent with AdministratorAccess secure?
You don't secure the agent — you contain it.

Layer 1

Account Isolation

Dedicated sandbox account. AWS account boundaries are the hardest security boundary in cloud. If everything goes wrong: close the account.

Layer 2

Profile-Based IAM

Three graduated profiles. personal_assistant can only call Bedrock. account_assistant is read-only with explicit denies. builder gets admin — but only in its sandbox.

Layer 3

Deny Policies

Explicit Deny statements on secrets, S3 objects, and Lambda code — even for read-only profiles. Deny overrides all Allows in IAM. Non-bypassable.

Layer 4

Security Services

GuardDuty, Security Hub, Inspector, IAM Access Analyzer, Config Recorder — all ON by default in the CFN template. The agent is watched from deploy minute one.

Layer 5

Network Containment

Single VPC. SSH disabled (127.0.0.1/32). No inbound ports. SSM is the only access path — and SSM logs every session to CloudTrail.

Layer 6

Full Audit Trail

No SSH keys = no unlogged access. Every API call via CloudTrail. Every SSM session recorded. Every resource change tracked by Config.

Layer 7

Disposability

Agent is cattle, not pets. Suspect compromise? Don't investigate — destroy and redeploy in 10 minutes. curl uninstall.lowkey.run | bash

"Like giving an intern their own AWS sandbox — they can break things, but only their things. And you have cameras everywhere."
Permission profiles

Graduated trust

Three profiles control what the agent can do. Start narrow, widen as you build confidence.

🔴 builder

Full Builder

AdministratorAccess — build apps, deploy infra, manage pipelines. Use in a dedicated sandbox account.

🟡 account_assistant

Read-Only Advisor

ReadOnlyAccess + Bedrock — cost analysis, architecture review, debugging. Can see everything, change nothing.

🟢 personal_assistant

Personal Assistant

Bedrock only — writing, research, coding help, daily tasks. No AWS infrastructure access whatsoever.

Key component

Bedrockify — the universal adapter

A local systemd daemon that exposes an OpenAI-compatible API on localhost:8090, proxying requests to Amazon Bedrock. Any agent that speaks the OpenAI protocol works on AWS without code changes.

OpenClaw integration

How Lowkey deploys OpenClaw

The OpenClaw pack installs a persistent AI gateway with memory, cron scheduling, multi-channel messaging, and Claude Code as a sub-agent.

Gateway

24/7 Daemon

Runs as systemd on port 3001. Always on, always reachable via SSM or messaging channels.

Memory

Persistent State

80GB data volume stores workspace, memory files, and session history across restarts.

Channels

Multi-Surface

Connect Discord, Telegram, or Slack. Agent responds to messages, runs cron jobs, posts updates.

Skills

Bootstraps

Post-deploy bootstraps configure alarms, coding standards, security protocols, and disk management.

Quick start

Three commands to try

# Full builder agent (OpenClaw — can create/modify/delete AWS resources) curl -sfL install.lowkey.run | bash -s -- -y --pack openclaw --profile builder # Read-only advisor (can see everything, change nothing) curl -sfL install.lowkey.run | bash -s -- -y --pack openclaw --profile account_assistant # Sandboxed personal assistant (NemoClaw — isolated in OpenShell sandbox) curl -sfL install.lowkey.run | bash -s -- -y --pack nemoclaw --profile personal_assistant

Requires: AWS CLI with admin access on a dedicated sandbox account. Works from CloudShell too.
