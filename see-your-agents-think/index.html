REASONING: FAIL

Agent ignored the "direct flights only" constraint. It picked the cheapest option overall — which happened to be direct. Next time it won't be.

CHAPTER 03

Building your
eval framework

7 habits of highly effective evals

01

Fast

Evals should run in seconds, not minutes or days. If you can't run them after every change, they won't get run at all. Treat eval speed like test speed — a 10-minute suite becomes shelfware. Target under 60 seconds for your core suite.

In practice: Run evals in CI on every PR. Use streaming and parallel execution. Cache expensive LLM-as-Judge calls where inputs haven't changed.

02

Quantifiable

Numbers, not feelings. Binary pass/fail is clearer and more consistent than 1–5 Likert scales. When you need nuance, use multiple binary evaluators rather than one multi-point scale.

In practice: "Did the agent book the correct restaurant?" (yes/no) is more actionable than "Rate the booking quality 1–5." Aggregate binary scores into pass rates over time.

03

Numerous

100+ test cases to catch variance. Single-case tests hide problems — multi-case evaluation reveals whether good performance is reliable or lucky. Agents are non-deterministic; statistical significance requires volume.

In practice: Start with 20–50 from real failures (bug tracker, support queue). Generate synthetic variants. Run each case multiple times to measure pass^k reliability, not just pass@1.

04

Explainable

Capture reasoning for both generation and scoring. When a score drops, you need to know why — not just that it dropped. LLM-as-Judge should output its reasoning alongside the verdict.

NOT EXPLAINABLE

Score: 2/5

Why did it fail? No idea. Back to guessing.

EXPLAINABLE

Score: 2/5

"Failed because the agent confirmed a booking that the tool returned as unsuccessful. Data accuracy: 1/5. The agent hallucinated a confirmation number."

05

Segmented

Evaluate every step independently. A single end-to-end score hides which step broke. Decompose into session, trace, and tool levels — problems invisible at one granularity become obvious at another.

In practice: Session-level: did the user achieve their goal? Trace-level: was each turn helpful? Tool-level: was each tool call justified and correctly parameterized?

EVALUATION GRANULARITY

Three levels of agent evaluation

Problems invisible at one level become obvious at another

Strands Evals blog

SESSION LEVEL

Did the user achieve their goal?

Examines the complete conversation end-to-end. Full interaction history, all tool executions, final outcome.

GoalSuccessRate

TRACE LEVEL

Was each turn helpful and faithful?

Turn-by-turn evaluation of individual responses. Judges quality, helpfulness, and harmfulness at each step.

Helpfulness · Faithfulness · Harmfulness

TOOL LEVEL

Was each tool call correct?

Evaluates individual tool invocations. Was the right tool selected? Were parameters correct? Was the call justified?

ToolSelectionAccuracy · ToolParameterAccuracy

06

Diverse

Cover every category of use case. Edge cases, adversarial inputs, multi-language, different user personas. Don't just test the happy path — the failures that reach production are always the ones you didn't think to test.

In practice: Segment test cases by category (simple/complex, single-tool/multi-tool, expected-success/expected-failure). Track pass rates per segment — a global 85% can hide a 20% in one category.

07

Traditional

Use traditional tools where they fit. Programmatic checks (exact match, regex, SQL queries, API assertions) are free, fast, and deterministic. Save LLM-as-Judge for what genuinely requires judgment.

In practice: "Did the agent call the right API?" → check the trace. "Did the response contain PII?" → regex. "Was the explanation helpful?" → LLM-as-Judge. Match the evaluator to the question.

COMMON MISCONCEPTIONS

Two things people get wrong about evals

MYTH 01

"The score is the truth"

If the score looks wrong, suspect the eval before the agent. Rigid grading and ambiguous tasks can tank scores artificially.

42% → fixed the eval → 95% (same agent)

MYTH 02

"We need hundreds of test cases first"

20–50 tasks from real failures is a great start. Your bug tracker and support queue are the mine.

The longer you wait, the harder it gets.
