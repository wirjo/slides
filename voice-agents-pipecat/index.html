Your total time budget

200ms

From the user finishing a sentence to your agent starting to respond. Chat agents get seconds. Voice agents don't get a second chance. If you miss the window, the user thinks the call dropped.

Where most implementations break

Three things most voice agents get wrong

01

Interruption

User starts talking mid-response. Do you stop, fade, or finish? The answer depends on context, and most agents don't know the difference

02

Silence

Is the user done talking or just thinking? Get this wrong and you either interrupt them or leave dead air

Latency

04 / 15
Problem 01

Interruption

The three responses

A

Stop

Cut audio immediately. Maximally responsive, but jarring on false positives

B

Fade

Taper out over 200ms. Natural-feeling, hides the latency

C

Finish

Complete the sentence before yielding. Right for ambient noise and back-channel cues

The decision logic

→

User corrected you → Stop immediately, discard pending audio

→

User affirmed ("yeah", "right") → Fade, continue response

→

User asked a new question → Stop, re-prompt LLM

→

Ambient noise or short utterance → Finish, don't interrupt yourself

Problem 02

Silence detection

Three regimes, each wrong in a different way

300ms

Too eager

Cuts the user off mid-thought. Feels robotic and doesn't respect natural pause rhythms

1.5s

Dead air

The user assumes the agent is broken or the call dropped. They'll speak again to fill the silence

VAD

Adaptive

Models speech rhythm and prosody to infer completion. The right answer

What adaptive VAD watches

→

Falling intonation at the end of a phrase

→

Sentence-completion syntax ("...so that's it", "...you know?")

→

Pause length relative to the user's own cadence in this call

→

Backoff: lengthen the timeout after a complex question

06 / 15
Problem 03

The 200ms budget

~40ms STT
~100ms LLM first token
~60ms TTS first chunk
= 200ms total

Sequential calls can't hit this

Wait for STT to finish, then call the LLM, then wait for TTS to start — and you're already at 400ms before any audio plays. Users notice.

Streaming pipeline is the answer

STT streams partial transcripts to the LLM. LLM streams first tokens to TTS. TTS synthesizes before the LLM finishes. Parallel, overlapping, pipelined.

The real hard problems

Audio engineering

Echo cancellation

Your agent's audio feeds back into the mic. Without AEC, the STT hears the agent's own voice and generates spurious transcripts that trigger false interruptions. Daily's WebRTC handles this — don't try to roll your own.

Backpressure

When the LLM is slower than real-time, audio chunks queue. If you don't manage flow control, you get garbled output, dropped frames, or a buffer that grows unbounded. Pipecat's pipeline handles this for you.

Audio chunking

You can't send one audio blob. STT models expect a specific frame size over WebSocket — typically 20ms chunks. Get the frame size or chunking cadence wrong and transcription accuracy drops significantly.

Pipecat — queue-per-stage · backpressure · frame routing · VAD events github.com/pipecat-ai/pipecat

No waiting between stages

Partial transcripts flow through the pipeline continuously. The LLM starts generating before the user finishes speaking. TTS starts before the LLM finishes.

VAD gates the LLM

The LLM only receives a UserStoppedSpeaking frame when VAD confirms turn end — not on every partial transcript.

Interruption cancels downstream

UserStartedSpeaking flushes the LLM output queue and cancels in-flight TTS frames. No extra glue code.

Pipecat · pipeline setup

Building the voice pipeline

from pipecat.pipeline.pipeline import Pipeline
from pipecat.services.deepgram import DeepgramSTTService, DeepgramTTSService
from pipecat.services.anthropic import AnthropicLLMService
from pipecat.transports.daily import DailyTransport
from pipecat.audio.vad.silero import SileroVADAnalyzer

transport = DailyTransport(
    room_url, token,
    params=DailyParams(
        audio_in_enabled=True,
        audio_out_enabled=True,
        vad_analyzer=SileroVADAnalyzer(params=VADParams(
            stop_secs=0.6          # wait 600ms of silence
        ))
    )
)

pipeline = Pipeline([
    transport.input(),         # WebRTC audio in + AEC
    DeepgramSTTService(api_key=...), # streaming STT
    AnthropicLLMService(model="claude-sonnet-4-6"),
    DeepgramTTSService(api_key=...), # streaming TTS
    transport.output()         # WebRTC audio out
])

What Pipecat handles for you

→

Each stage is a processor that reads from an input queue and writes to an output queue — backpressure is built in

→

STT partial transcripts flow into the LLM immediately — no waiting for a final transcript

→

LLM token stream feeds directly into TTS — audio starts before generation completes

→

Swap any stage without touching the rest — same interface for all STT, TTS, and LLM providers

Silence detection · implementation

Implementing VAD in Pipecat

from pipecat.audio.vad.silero import SileroVADAnalyzer, VADParams
from pipecat.audio.vad.vad_analyzer import VADState

# Tune the three key parameters
vad = SileroVADAnalyzer(params=VADParams(
    confidence=0.7,    # speech probability threshold
    start_secs=0.2,    # speech detected after 200ms
    stop_secs=0.6,     # speech ended after 600ms silence
))

# Hook VAD state changes
async def on_user_speech_start(frame):
    # Cancel in-flight TTS if user starts talking
    await pipeline.cancel_tts()

async def on_user_speech_stop(frame):
    # Trigger LLM only when VAD confirms turn end
    await llm.process_frame(transcript_frame)

transport.on("vad_started", on_user_speech_start)
transport.on("vad_stopped", on_user_speech_stop)

Silero VAD vs energy-based VAD

→

Energy thresholds trigger on keyboard noise, background TV, HVAC. Silero is a neural model that distinguishes speech from non-speech

→

stop_secs is your primary tuning lever. 0.6s works well conversationally. Increase to 0.8–1.0s for complex questions where users pause to think

→

Silero runs locally at ~5ms per frame — no round-trip latency, no API cost, no data leaving the server

Research · beyond VAD

Smart turn detection

The problem with VAD alone

→

VAD tells you when audio energy stops. It can't tell you if the user's thought is complete. "I want to book a flight to... Sydney" — VAD fires after the pause, before Sydney

→

Listing items ("I need milk, eggs, uh...") produces repeated false turn-ends. The agent starts responding to an incomplete list

→

Questions with rising intonation followed by a long pause ("Have you ever tried... going to the gym at 6am?") confuse fixed-threshold detectors

What the research says

01

Turn-GPT (2022)

LLM-based turn completion predictor trained on dialog corpora. Predicts whether a transcript is a complete utterance. ~120ms inference on GPU

02

Smart Turn (Daily, 2024)

Production model trained on voice agent conversations. Combines audio features and partial transcript. Shipped in Daily's SDK as a drop-in replacement for Silero

03

Pipecat's SmartTurnAnalyzer

Same interface as SileroVADAnalyzer — swap SmartTurnAnalyzer() in the DailyParams and get semantic turn detection with no pipeline changes

Key takeaways

200ms. Interruption. Silence.
These are engineering problems.

Everything must stream in parallel. STT into LLM into TTS. Sequential calls will never hit the latency budget.

Let Daily handle WebRTC and echo cancellation. The audio transport layer is not where you want to spend your debugging time.

Let Pipecat handle pipeline orchestration and backpressure. Write the interruption logic and the silence model — not the plumbing.
