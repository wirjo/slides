THE CHALLENGE

Training and serving large models requires solving multiple hard distributed systems problems simultaneously

Training challenges

  • Models exceed single-GPU memory (405B params = ~810 GB in FP16)
  • Data, model, tensor, pipeline, context, and expert parallelism strategies
  • Gradient synchronization across hundreds of GPUs via AllReduce/AllGather
  • Checkpointing and fault-tolerant resumption at scale
  • Network topology-aware job placement for optimal bandwidth

Inference challenges

  • KV-cache memory grows linearly with sequence length and batch size
  • Prefill is compute-bound, decode is memory-bandwidth-bound
  • Disaggregated architectures need low-latency cross-node transfers
  • Expert parallelism for MoE models across multiple nodes
  • Continuous batching and request scheduling at high throughput
Model parameter sizes: Meta Llama 3.1 (405B), DeepSeek-V3 (671B MoE)

INFRASTRUCTURE

Cluster topology: control plane, workers, and networking

Control plane (AWS-managed)

  • EKS control plane is fully managed by AWS (no customer nodes)
  • 1:1 mapping between one EKS cluster and one HyperPod cluster
  • API server, etcd, and scheduler run in AWS-managed VPC
  • Accessible via kubectl, HyperPod CLI, or AWS Console
  • No load balancer required for training workloads (pod-to-pod only)

Worker nodes (HyperPod-managed)

  • GPU instances (P5, P5e, P4d) placed in private subnets
  • EFA-enabled networking with placement groups for optimal topology
  • HyperPod manages lifecycle: provisioning, health checks, replacement
  • VPC, subnets, security groups, IAM created via CloudFormation
  • FSx for Lustre or S3 mounted for shared storage

Topology-aware scheduling: P5/P5e jobs are packed onto a single NVLink/NVSwitch domain first (900 GB/s), spreading to other nodes over EFA (3200 Gbps) only when local capacity is exhausted. UltraServers extend this to 72-GPU NVL72 chassis.

INFRASTRUCTURE

Unified cluster: the 1:1 EKS mapping and how to bring existing workloads

The 1:1 architecture

  • One HyperPod cluster maps to exactly one EKS control plane
  • HyperPod creates and owns the EKS cluster (via CloudFormation)
  • All GPU worker nodes are HyperPod-managed with health checks and auto-recovery
  • You get full kubectl and RBAC access to the underlying EKS cluster
  • Non-GPU workloads (APIs, data pipelines) can be scheduled on the same cluster using separate node groups

Running mixed workloads

  • Add standard EKS managed node groups (CPU) alongside HyperPod GPU nodes
  • Kubernetes namespaces isolate ML workloads
  • Kueue governs GPU job admission; standard K8s pods schedule normally via the default scheduler
  • Shared cluster networking, service discovery, and IAM IRSA
  • Single observability stack (CloudWatch, Prometheus) for all workloads

Key principle: HyperPod maintains a 1:1 mapping between one EKS cluster and one HyperPod cluster. You can either let HyperPod create the EKS cluster via CloudFormation, or associate HyperPod with an existing EKS cluster. Either way, you can add your own non-HyperPod node groups and deploy any Kubernetes workload alongside the managed GPU nodes.

NETWORKING

Intra-node and inter-node interconnects for distributed AI

Intra-node refers to communication between GPUs within a single instance (e.g. 8 GPUs in one p5.48xlarge). Inter-node refers to communication between separate instances across the network.

Intra-node: NVLink and NVSwitch

  • NVLink 4 (P5/P5e): 900 GB/s bidirectional between all 8 GPUs within a single instance
  • NVSwitch provides full-mesh GPU-to-GPU connectivity (no CPU hop)
  • Used for tensor parallelism, where layer shards communicate every forward/backward pass
  • Coming: NVLink 5 on P6/UltraServers with higher per-link bandwidth across 72 GPUs

Inter-node: Elastic Fabric Adapter (EFA)

  • P5/P5e: 3200 Gbps (400 GB/s) per instance via multiple EFA interfaces
  • OS-bypass RDMA via libfabric: data moves directly from application memory to the network card without going through the operating system kernel, dramatically reducing latency
  • Used for data parallelism (AllReduce gradients), pipeline parallelism (activations), and KV-cache transfer (NIXL)
  • Placement groups co-locate instances on the same network spine, minimizing hops and latency between nodes

Why this matters: Topology-aware scheduling in HyperPod packs tensor-parallel groups onto a single NVLink domain (900 GB/s) and only crosses the EFA boundary (400 GB/s) for data parallelism. Choosing the wrong parallelism-to-topology mapping can reduce training throughput by 2-5x.

TRAINING FUNDAMENTALS

Parallelism strategies for large model training

Data parallelism
Each GPU holds a full model copy and processes a different data shard. Gradients are synchronized via AllReduce. Simple but limited by model size fitting in one GPU.
Model/tensor parallelism
Individual layers are split across GPUs (e.g. attention heads on separate devices). Reduces per-GPU memory. Requires high-bandwidth interconnect (NVLink).
Pipeline parallelism
Different layers are assigned to different GPUs in stages. Micro-batches flow through the pipeline. Balances memory and compute but introduces bubble overhead.

Context parallelism

  • Splits long sequences across GPUs
  • Each GPU processes a portion of the sequence
  • Critical for long-context models (128K+ tokens)

Expert parallelism (training and inference)

  • Different experts reside on different GPUs
  • Router sends tokens to relevant experts
  • Used in training (DeepSeek-V3, Mixtral) and inference (llm-d)
Expert parallelism applies to both training (distributing experts across GPUs) and inference (llm-d routes to expert-specific nodes)

INFERENCE FUNDAMENTALS

Why disaggregate prefill from decode?

Prefill phase

  • Processes the full input prompt in one forward pass
  • Compute-bound: high FLOPS utilization
  • Generates the KV-cache for all input tokens
  • Benefits from high GPU compute density
  • Latency is O(input_length)

Decode phase

  • Generates one token at a time, autoregressively
  • Memory-bandwidth-bound: reads full KV-cache per token
  • Benefits from high memory bandwidth (HBM3e)
  • Continuous batching improves throughput
  • Latency is O(output_length x KV_cache_reads)

Prefill and decode have opposite hardware requirements. Running them on separate node pools (disaggregated) lets you optimize hardware allocation independently. For MoE models, expert parallelism routes tokens to expert-specific nodes during both prefill and decode, further benefiting from disaggregation.

INFERENCE FUNDAMENTALS

When to disaggregate inference and what characteristics matter

Disaggregation benefits you when

  • Long input contexts (32K+ tokens): prefill dominates end-to-end latency and benefits from dedicated compute
  • High request concurrency: independent scaling of prefill and decode pools avoids head-of-line blocking
  • Asymmetric traffic patterns: bursty prefill load with steady decode streams
  • MoE models: expert parallelism routes to specialized nodes, reducing memory duplication
  • Cost optimization: right-size instance types per phase (compute-dense vs memory-bandwidth-dense)

Co-located (monolithic) is fine when

  • Short prompts (<4K tokens): prefill is cheap and KV-cache transfer overhead outweighs the benefit
  • Low throughput requirements: single node handles both phases without saturation
  • Latency-sensitive, short-output tasks: KV-cache network hop adds P99 tail latency
  • Small models that fit in a single GPU: no cross-node communication needed
  • Simpler operations: fewer components to monitor and debug

Rule of thumb: If your P50 input is over 8K tokens and you need to serve 100+ concurrent requests, disaggregation typically yields 2-3x better throughput per dollar. Below that, monolithic vLLM is simpler and sufficient.

INFERENCE STACK

Understanding llm-d: Kubernetes-native inference orchestration

llm-d is an open-source inference orchestration framework built on vLLM. It adds intelligent scheduling, disaggregated serving, and tiered caching to solve LLM serving at scale. llm-d integrates low-level runtime components from NVIDIA Dynamo, including NIXL for KV cache transport and the Dynamo Planner for GPU resource management.

1
Inference Gateway (IGW)
Entry point for all inference traffic. Extends the Kubernetes Gateway API with inference-aware routing via the Endpoint Picker Protocol (EPP) — a standard K8s interface that lets llm-d plug its routing logic into the gateway without a custom proxy
2
KV-cache-aware scheduler
llm-d's EPP implementation — makes decisions like "this request has a prefix matching the KV cache on pod-7, send it there." Avoids redundant prefill by routing to replicas that already hold the relevant context
3
Pool orchestration sidecar
Manages the lifecycle of disaggregated prefill and decode GPU pools via a sidecar on each node, coordinating request handoff and KV cache transfers between pools
4
NIXL transport (via NVIDIA Dynamo)
Moves the KV cache directly from one GPU's memory to another GPU's memory across nodes, without copying through the CPU or operating system. Uses EFA for high-speed network transfer, keeping latency minimal during prefill-to-decode handoff
5
Managed tiered KV cache
L1 per-node (CPU memory) + L2 distributed cluster-wide (terabyte-scale, GPU-aware zero-copy). Offloads KV cache beyond GPU VRAM — repeat prompts hit the cache and skip prefill entirely. Enabled via CRD on the HyperPod Inference Operator.
6
Workload Variant Autoscaler (WVA)
Monitors request rates and GPU utilization per pool, then emits target replica counts as Prometheus metrics. KEDA consumes these signals and executes the actual pod scaling. A surge in long-context prompts scales up prefill; chatbot traffic scales up decode.
github.com/llm-d/llm-d | github.com/ai-dynamo/dynamo | AWS container: ghcr.io/llm-d/llm-d-aws

INFERENCE STACK

HyperPod Inference Operator: the managed control plane

A Kubernetes operator that abstracts inference complexity into a single Custom Resource Definition (CRD). Apply a YAML manifest and HyperPod handles model deployment, intelligent routing, tiered KV cache, autoscaling, and observability.

What the operator manages

Model deployment: Downloads weights from S3, FSx, or HuggingFace. Spins up vLLM pods with the right GPU resources.

Intelligent routing: Deploys a router with three strategies — prefix-aware (default), KV-cache-aware, or round-robin.

Managed tiered KV cache: L1 per-node (CPU memory) + L2 distributed (terabyte-scale, GPU-aware zero-copy). No infra to provision.

Autoscaling: Integrates Karpenter for scale-to-zero and back. Metrics from CloudWatch or Prometheus.

Endpoint registration: Automatically registers a SageMaker endpoint or provisions an ALB. Deleting the CRD cleans up everything.

Performance impact: routing + KV cache combined

Llama 3.1 70B Instruct on 7x p5.48xlarge. Intelligent routing avoids redundant prefill by sending requests to replicas that already hold the relevant KV cache. The L1/L2 tiered cache retains computed blocks so repeated context (system prompts, multi-turn history) skips prefill entirely.

OPERATIONS

Global capacity orchestration: routing across regions

The challenge

  • GPU capacity (P5, P5e) is scarce and unevenly distributed across regions
  • HyperPod clusters are region-bound (1:1 EKS mapping; EKS is a regional service)
  • A single region may not satisfy all your GPU demand for training or inference
  • Data gravity: training datasets in S3/FSx are regional, so cross-region training incurs transfer costs

Multi-region pattern

  • Deploy separate HyperPod EKS clusters in multiple regions (e.g. us-east-1, us-west-2, eu-west-1)
  • Use an AI gateway or routing layer to distribute inference requests across regional endpoints
  • Route 53 latency-based routing or health-check failover between regional llm-d endpoints
  • For training: job dispatcher routes to whichever region has available capacity

AI gateway pattern: A routing layer (API Gateway, CloudFront, or a custom gateway) that is capacity-aware and directs inference requests to whichever regional HyperPod cluster has available GPU resources. llm-d handles intra-cluster routing; the AI gateway handles inter-region routing. This is a customer-built architectural pattern, not a built-in HyperPod feature.

CHOOSING YOUR PATH

HyperPod managed vs individual components on bare EKS

Use HyperPod when you want

  • Validated, pre-integrated stack with health management
  • Auto-resume for long-running training jobs
  • Topology-aware scheduling without custom logic
  • Built-in governance for multi-team GPU clusters
  • Fast path to production with recipes for popular models
  • Reduced undifferentiated heavy lifting on infra

Cherry-pick components when you need

  • llm-d for inference orchestration on your own EKS cluster
  • vLLM as your serving engine (works standalone)
  • HyperPod Recipes for training configs (Slurm or K8s)
  • NIXL for KV-cache transfers in custom architectures
  • Kueue for job governance (K8s native, no HyperPod required)
  • Full control over driver versions, AMIs, and scheduling

Key takeaway: HyperPod's value is in the integration and operational management. Every component listed in this deck has a standalone OSS path if you prefer to assemble the stack yourself.
